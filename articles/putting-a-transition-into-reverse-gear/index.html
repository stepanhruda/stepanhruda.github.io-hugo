Part 2: Putting a transition in reverse gear

Putting the easy into easy transitioning

Stepan Hruda on 02 June 2014

This article is part of the Transition with STPTransitions series.

In the first article, we created a pretty snazzy transition when pushing view controllers. It'd be cool if we also animated in reverse when going back.

STPTransiton has a reverseTransition property, which you can specify when pushing. This way, any pushed view controller is agnostic of how it was presented and doesn't have to worry about anything when going back.

STPTransition *transition = [STPRotateFadeTransition new];
transition.reverseTransition = [STPRotateFadeTransition new];
[self.navigationController pushViewController:secondViewController
                              usingTransition:transition];

Note: STPTransition is a representation of a one-time transition between two view controllers. Don't reuse the instances, simply create new ones. It's cheap!

Wrong way

Unfortunately, only specifying a reverse transition is not enough right now. This is what happens when you simply add a reverse transition.

While looking cool, this is not what you usually want.

Of course it goes the wrong way! We never told anyone what the animation should look like when going in the other direction. It's easy enough to spill the beans on that â€“ we can modify our transition class to account for both going forwards and backwards.

Pro tip: We could of course create a separate transition class for the other direction. In most cases, the code is going to be so similar to the original class that one STPTransition can easily handle both directions and is much easier to maintain.

Naming

Before we make the change, let's make sure we are clear on the naming: toView is always the view we're trying to get to. fromView is the view we're starting with.

first controller       second controller

                pushing
     fromView     ->       toView

                popping
      toView      <-      fromView

Alpha

We don't need to change anything about the alphas of the views at all: the newly appearing toView goes from 0->1, the disappearing fromView goes 1->0 when going in either direction. Not having to write any code is almost as rewarding as deleting code!

Rotation

When pushing, fromView was rotating to the left, now we need it to rotate to the right. To figure out which way we're going, we can leverage the isReversed property provided by STPTransition and switch the rotation if it's YES.

toView.layer.transform = !self.isReversed ? [self rotatedRightToX:offsetX] : [self rotatedLeftToX:offsetX];

Vice versa for the toView. We need it to go to the right in case we're going in reverse.

fromView.layer.transform = !self.isReversed ? [self rotatedLeftToX:offsetX] : [self rotatedRightToX:offsetX];

Done and done

That's it, we rotate in the other direction when going back. Those are all the changes we really need to do for our transition to work in both directions.

It works both for the default < Back button and for our custom one more time button.

Decoupling

Take a look at the implementation of the one more time button's action. It's so dumb it could be on reality TV. This controller doesn't have any transition-specific code at all!

- (void)backButtonTapped:(UIButton *)sender {
    [self.navigationController popViewControllerAnimated:YES];
}

Pro tip: The controller still knows it has a navigation controller. If we wanted to make it totally decoupled, we could implement an onBackAction block property. The presenting controller would then supply the block for the dimissal.

Choosing your pop transition

Sometimes you might not want to be surfing the decoupling wave. You can hard-choose the popping transition using -popViewControllerUsingTransition:

[self.navigationController popViewControllerUsingTransition:[STPSlideUpTransition new]];

This will result in a different transition when going back.

Pro tip: If we pass in nil, the navigation controller will use the default iOS pop transition.

Updated transition's code for reference:

// STPRotateFadeTransition.h

@interface STPRotateFadeTransition : STPTransition

@end

// STPRotateFadeTransition.m

#import "STPRotateFadeTransition.h"

@implementation STPRotateFadeTransition

- (NSTimeInterval)transitionDuration {
    return 0.5f;
}

- (void)animateFromView:(UIView *)fromView
                 toView:(UIView *)toView
        inContainerView:(UIView *)containerView
    executeOnCompletion:(void (^)(BOOL))onCompletion {
    toView.alpha = 0.0f;

    CGFloat offsetX = CGRectGetWidth(containerView.bounds) / 2.5f;

    toView.layer.transform = !self.isReversed ? [self rotatedRightToX:offsetX] : [self rotatedLeftToX:offsetX];

    [containerView addSubview:toView];

    [UIView animateWithDuration:self.transitionDuration
                     animations:
     ^{
         fromView.alpha = 0.0f;
         toView.alpha = 1.0f;

         fromView.layer.transform = !self.isReversed ? [self rotatedLeftToX:offsetX] : [self rotatedRightToX:offsetX];
         toView.layer.transform = CATransform3DIdentity;
     }
                     completion:
     ^(BOOL finished) {
         onCompletion(finished);

         fromView.alpha = 1.0f;
         toView.alpha = 1.0f;

         fromView.layer.transform = CATransform3DIdentity;
         toView.layer.transform = CATransform3DIdentity;
     }];
}

- (CATransform3D)rotatedLeftToX:(CGFloat)offsetX {
    CATransform3D rotateNegatively = CATransform3DMakeRotation(-M_PI_2, 0, 1, 0);
    CATransform3D moveLeft = CATransform3DMakeTranslation(-offsetX, 0, 0);
    return CATransform3DConcat(rotateNegatively, moveLeft);
}

- (CATransform3D)rotatedRightToX:(CGFloat)offsetX {
    CATransform3D rotatePositively = CATransform3DMakeRotation(M_PI_2, 0, 1, 0);
    CATransform3D moveRight = CATransform3DMakeTranslation(offsetX, 0, 0);
    return CATransform3DConcat(rotatePositively, moveRight);
}

@end
